Agent 记忆机制深度解析
总览：为什么 Agent 需要记忆？
LLM 本身是无状态的——每次调用都是一次全新的对话，它不记得上一次跟你说了什么。记忆机制就是解决"让 Agent 记住东西"的问题。

没有记忆的 Agent:
  用户: "帮我分析茅台"     → Agent 分析完毕
  用户: "和五粮液对比呢？"  → Agent: "对比什么？你之前分析了什么？我不知道"
  用户: "上次你说PE是31倍" → Agent: "我说过这话？我没有任何记忆"
有记忆的 Agent:
  用户: "帮我分析茅台"     → Agent 分析完毕，记住分析结果
  用户: "和五粮液对比呢？"  → Agent: 从记忆中取出茅台分析 + 获取五粮液数据 → 对比
  用户: "上次你说PE是31倍" → Agent: 从记忆中确认 → "是的，当时PE是31倍，现在..."
人类记忆的类比
Agent 的三种记忆机制直接类比人类认知科学中的记忆分类：

一、短期记忆（上下文窗口）
1.1 核心原理
短期记忆就是 LLM 的上下文窗口（Context Window）——每次 API 调用时传入的 messages 数组。LLM 能"看到"并"记住"的所有信息，都在这个窗口里。

本质：短期记忆不是真正的"记忆"，而是"把对话历史塞进输入"。LLM 每次调用都是无状态的，它之所以"记得"之前的对话，只是因为你把之前的对话完整地传了进去。

1.3 上下文窗口管理策略
当对话足够长，消息总量会超过窗口限制。需要上下文管理策略来决定保留什么、丢弃什么。

策略 1: 滑动窗口 (Sliding Window)
─────────────────────────────────────────────────
保留最近 N 轮对话，丢弃更早的。
  轮次 1  轮次 2  轮次 3  轮次 4  轮次 5  轮次 6
  ┌────┐  ┌────┐  ┌────┐  ┌────┐  ┌────┐  ┌────┐
  │ M1 │  │ M2 │  │ M3 │  │ M4 │  │ M5 │  │ M6 │
  └────┘  └────┘  └────┘  └────┘  └────┘  └────┘
  窗口大小=4 时:
  当前窗口: [M3, M4, M5, M6]
  丢弃:     [M1, M2]
  ✅ 简单高效
  ❌ 丢失的早期信息可能很重要
     ("用户第一轮说了投资目标是保本，后面全忘了")
策略 2: 摘要压缩 (Summarization)
─────────────────────────────────────────────────
对早期对话生成摘要，用摘要替代原始消息。
  ┌──────────────────────────────────────────┐
  │  Summary (由 LLM 生成):                   │
  │  "用户询问了茅台的估值分析。茅台当前       │
  │   PE=31.5x，五粮液PE=22.8x，溢价38%。     │
  │   用户的投资偏好是稳健型，关注估值安全     │
  │   边际。"                                  │
  │                                    ~200 tokens (原文 ~5000)
  └──────────────────────────────────────────┘
  +
  ┌────┐  ┌────┐  ┌────┐
  │ M4 │  │ M5 │  │ M6 │   最近的完整消息
  └────┘  └────┘  └────┘
  ✅ 保留关键信息，大幅压缩 token
  ❌ 摘要可能丢失细节
  ❌ 生成摘要本身需要一次 LLM 调用（额外成本和延迟）
策略 3: Token 感知的智能裁剪
─────────────────────────────────────────────────
按重要性评分，优先保留重要消息。
  重要性评估:
  ┌──────────────────────────────────────────────────────┐
  │  M1: [User] "我的投资目标是保本增值"      重要性: 9  │ ← 保留
  │  M2: [Asst] "好的，我理解了"              重要性: 2  │ ← 丢弃
  │  M3: [User] "分析茅台"                    重要性: 5  │ ← 看空间
  │  M4: [Tool] {大量JSON数据}                重要性: 3  │ ← 压缩
  │  M5: [Asst] "PE=31.5x，结论:偏高"         重要性: 8  │ ← 保留
  │  M6: [User] "对比五粮液"                  重要性: 7  │ ← 保留
  └──────────────────────────────────────────────────────┘
  保留: [M1, M5, M6] + 最新轮次
  压缩: M4 (只保留关键字段)
  丢弃: M2, M3
  ✅ 最优的信息密度
  ❌ 重要性评估本身有成本和不确定性

1.5 局限性
┌────────────────────────────────────────────────────────┐
│  短期记忆的根本限制                                     │
│                                                        │
│  1. 容量有限                                           │
│     128K tokens ≈ 一本小说的篇幅                        │
│     看似很多，但金融分析中的原始数据                     │
│     （财报、K线、研报）很容易撑满                       │
│                                                        │
│  2. 线性成本                                           │
│     对话越长 → 每次 API 调用传的 token 越多             │
│     → 成本线性增长、延迟线性增长                        │
│     一个 100K token 的请求比 10K 贵 10 倍               │
│                                                        │
│  3. 注意力衰减 ("Lost in the Middle")                  │
│     LLM 对上下文窗口中间位置的信息                      │
│     注意力显著弱于开头和结尾                             │
│     = 重要信息放在中间可能被"忽视"                      │
│                                                        │
│  4. 跨会话遗忘                                         │
│     用户关闭浏览器 → 新会话 → 一切重新开始              │
│     没有持久化机制就没有"长期记忆"                      │
│                                                        │
└────────────────────────────────────────────────────────┘
二、长期记忆（向量检索）
2.1 核心原理
长期记忆解决短期记忆的两个根本问题：容量有限 和 跨会话遗忘。

核心思路：把需要"记住"的信息存到外部数据库，需要用时通过 语义检索 取回相关片段，注入到当前上下文窗口。

短期记忆: 所有信息塞进 LLM 输入 → 容量受限
长期记忆: 信息存在外部 → 按需检索 → 只取相关的塞进 LLM 输入 → 容量无限

2.5 记忆衰减与遗忘机制
人类会遗忘，Agent 也应该有遗忘机制。不加遗忘的记忆系统会越来越嘈杂。

┌────────────────────────────────────────────────────────────────┐
│                    记忆衰减模型                                  │
│                                                                │
│  灵感来源: 心理学 Ebbinghaus 遗忘曲线                           │
│                                                                │
│  记忆强度                                                      │
│  ▲                                                             │
│  │ ██                                                          │
│  │ ██                                                          │
│  │ ██ ▓▓                                                       │

│  │ ██ ▓▓ ▓▓ ░░                                                 │
│  │ ██ ▓▓ ▓▓ ░░ ░░                                              │
│  │ ██ ▓▓ ▓▓ ░░ ░░ ░░                                           │
│  │ ██ ▓▓ ▓▓ ░░ ░░ ░░ ·· ·· ·· ··                               │

│   创建                                 现在                    │
│                                                                │
│  衰减公式:                                                     │

│  其中:                                                         │
│  - importance: 初始重要性 (用户偏好=10, 普通事实=3)             │

│                                                                │
│  ┌──────────────────────────────────────┐                      │
│  │  不同类型的衰减速率                   │                      │

│  │  投资决策:     λ=0.01   (半衰期69天)  │  季度级衰减          │
│  │  分析结论:     λ=0.03   (半衰期23天)  │  月级衰减            │
│  │  行情数据:     λ=0.1    (半衰期7天)   │  周级衰减            │
│  │  临时对话细节: λ=0.5    (半衰期1.4天) │  很快遗忘            │

│                                                                │
│  定期清理:                                                      │
│  strength < threshold → 标记为"弱记忆" → 归档/删除              │
└────────────────────────────────────────────────────────────────┘
2.6 局限性
1. 检索质量瓶颈
   向量相似度 ≠ 任务相关性
   "茅台酒好喝吗" 和 "茅台投资价值" 语义相近，但需求完全不同
2. 上下文污染
   检索到的记忆可能与当前问题无关，塞进上下文后反而误导 LLM
3. 一致性问题
   同一个事实可能被存储多次（不同时间点），且相互矛盾
   "3月分析: PE=28x" vs "4月分析: PE=31.5x" → 哪个是最新的？
4. 隐私与合规
   金融场景中，用户的投资偏好、持仓信息属于敏感数据
   需要严格的隔离（不同用户的记忆绝对不能串）和加密
三、工作记忆（Scratchpad）
3.1 核心原理
工作记忆是 Agent 执行复杂任务时的 临时推理空间——类似于人类做数学题时的草稿纸。

它与短期记忆和长期记忆的本质区别：

短期记忆: 被动的——把对话历史喂给 LLM，LLM 自己去"看"
长期记忆: 被动的——从数据库检索，注入到上下文，LLM 自己去"看"
工作记忆: 主动的——Agent 自己维护的结构化笔记，
                  主动决定写什么、读什么、更新什么、删什么

关键优势:
┌─────────────────────────────────────────────────────┐
│  没有 Scratchpad: 每一步都要传完整的对话历史         │
│  Step 5 的输入 = System + M1 + M2 + ... + M10       │
│  → 50K tokens 的输入，充满冗余                      │
│                                                     │
│  有 Scratchpad: 每一步只传 System + 当前问题 + Pad   │
│  Step 5 的输入 = System + 问题 + Scratchpad(2K)     │
│  → 5K tokens 的输入，信息密度极高                   │
│                                                     │
│  = 更省 token、更低延迟、LLM 推理质量更高           │
│    (因为"干扰信息"更少)                             │

五、面试高频问题
Q1: "Lost in the Middle" 问题是什么？怎么解决？

LLM 对上下文窗口中间位置的信息注意力显著弱于开头和结尾（Liu et al., 2023 实验验证）。解决方案：(1) 将最重要的信息放在开头（System Prompt）和结尾（最近的消息） (2) 中间位置放辅助信息 (3) 对长上下文做摘要压缩，减少中间段的长度 (4) 使用支持更好长上下文注意力分布的模型架构（如 Gemini 的 Ring Attention）。

Q2: 向量检索 vs 全文检索，什么时候用哪个？

向量检索捕获语义（"公司盈利能力" ≈ "净利润率"），全文检索捕获精确关键词（"600519" 只能精确匹配）。金融场景中两者互补：用户问"茅台赚钱能力如何"用向量检索；用户问"600519 的 PE 是多少"用全文检索。生产系统通常用 Hybrid Search（向量 + BM25 融合）。

Q3: Scratchpad 为什么不直接用对话历史代替？

三个原因：(1) 信息密度——对话历史中 80% 是冗余的客套话和工具调用的原始 JSON，Scratchpad 只保留精炼的关键值 (2) 结构化——对话是线性的，Scratchpad 是按功能分区的（任务/数据/结论），LLM 更容易定位信息 (3) token 效率——10 步任务的完整对话可能 30K tokens，Scratchpad 压缩到 2K tokens，省 10 倍成本。

Q4: 金融 Agent 的记忆系统有什么特殊的安全要求？

(1) 用户隔离：用户 A 的投资偏好/持仓绝不能出现在用户 B 的记忆检索结果中，向量数据库必须按 user_id 严格过滤 (2) 内幕信息防护：如果 Agent 服务多家上市公司，一家的非公开信息不能通过记忆系统泄露给另一家的分析 (3) 审计追溯：所有记忆的写入/读取操作必须有日志，满足金融监管的可追溯要求 (4) 遗忘权：用户要求删除数据时，必须从向量数据库中物理删除（不只是标记删除）。
